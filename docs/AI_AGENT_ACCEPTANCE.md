# English Assist AI Agent 落地验收文档

## 1. 文档目标

本文档用于从 AI agent 落地应用的角度，对 `English Assist` 进行验收定义。

验收重点不是“是否接入了模型”，也不是“是否存在聊天功能”，而是：

- 是否形成了多个 agent 的职责分工
- 是否形成了 agent 之间的协同闭环
- 是否能围绕真实学习任务持续工作
- 是否能向教师和学生输出可解释、可执行、可复盘的结果

本文档尤其突出一个核心特点：

**这个应用应被定义为“多个教学 agent 协调工作”的系统，而不是单一聊天机器人。**

---

## 2. 产品定位

### 2.1 定位说明

`English Assist` 是一个面向初高中英语听口训练场景的 AI 学习系统。

它的目标不是提供一个泛化问答助手，而是围绕教材同步学习，构建多个 agent 协同工作的教学闭环，覆盖：

- 学习诊断
- 学习规划
- 口语训练
- 发音评测
- 错题复盘
- 教师洞察

### 2.2 推荐对外表述

建议对外使用如下表述：

> English Assist 是一个面向英语教学场景的多 agent 协同学习系统。它通过学习编排 agent、口语陪练 agent、评测反馈 agent 与教师洞察 agent 的协同工作，把教材学习、口语训练、错题复盘和教学管理连接成一条完整闭环。

---

## 3. 多 Agent 架构定义

为便于评审与验收，建议将系统中的 AI 能力拆分为以下四类 agent。

### 3.1 学习编排 Agent

职责：

- 读取学生当前教材进度
- 读取错题、发音薄弱项、完成情况
- 决定“下一步最应该做什么”
- 生成今日学习任务顺序

产出：

- 今日任务推荐
- 阶段学习路径
- 每日/每单元学习计划

### 3.2 口语陪练 Agent

职责：

- 围绕当前教材单元发起对话训练
- 根据年级与难度控制表达要求
- 在不打断流畅度的前提下进行纠错
- 生成对话总结和改进建议

产出：

- 口语对练过程
- 纠错反馈
- 表达优化建议

### 3.3 评测反馈 Agent

职责：

- 汇总听写、朗读、发音评测结果
- 识别典型错误类型
- 提炼复习优先级
- 把弱项转化为可执行的训练建议

产出：

- 发音薄弱词
- 错题归因
- 复习建议
- 阶段性评测报告

### 3.4 教师洞察 Agent

职责：

- 汇总班级或个人学习数据
- 标记风险学生
- 生成教师干预建议
- 输出阶段报告与周报

产出：

- 班级趋势
- 风险名单
- 教师建议
- 可讲给家长或管理者的简报材料

## 4. 多 Agent 协调工作机制

本项目的亮点不应只强调“有多个 agent”，而应强调“多个 agent 能否协同工作”。

### 4.1 协同链路

理想链路如下：

1. 学习编排 Agent 读取当前学习状态
2. 决定学生先完成哪一类任务
3. 口语陪练 Agent 执行对应对练任务
4. 评测反馈 Agent 对结果进行归因和总结
5. 学习编排 Agent 根据结果调整后续任务
6. 教师洞察 Agent 汇总结果并输出教师视角建议

### 4.2 协同价值

相比单一聊天机器人，多 agent 协调工作的价值体现在：

- 每个 agent 目标明确，职责清晰
- 能围绕一条学习主线持续工作
- 能把输入、决策、执行、反馈连接起来
- 能同时服务学生侧与教师侧
- 更容易被解释、评估和运营

## 5. 验收范围

本次验收聚焦以下四类能力。

### 5.1 学生侧学习闭环

验收是否已经形成：

- 教材进入
- 学习任务推荐
- 练习执行
- 错误识别
- 结果反馈
- 再次推荐

### 5.2 多 Agent 协同链路

验收是否已经形成：

- 编排 agent 是否能驱动其他 agent
- 口语与评测 agent 是否能共享上下文
- 反馈结果是否能回流到下一轮推荐
- 教师端是否能消费学生端 agent 输出

### 5.3 教学可解释性

验收是否已经形成：

- 为什么推荐当前任务
- 为什么判定为薄弱项
- 为什么给出该条复习建议
- 教师为什么应该干预该学生

### 5.4 可持续运行能力

验收是否已经形成：

- agent 是否能跨会话连续工作
- agent 是否具备学习记忆
- agent 是否能随着结果动态调整策略

## 6. 核心验收标准

以下采用三级标准：

- `A 类`：必须通过，否则不能称为 AI agent 落地应用
- `B 类`：建议通过，否则可演示但不够扎实
- `C 类`：加分项，通过后更适合对外推荐

### 6.1 A 类标准：必须项

#### A1. 必须存在清晰的多 agent 分工

通过标准：

- 系统能清楚说明至少 3 类 agent 的职责边界
- 每个 agent 有明确输入、处理逻辑和输出结果
- 系统不是只有一个统一聊天入口承担所有能力

#### A2. 必须形成 agent 协同闭环

通过标准：

- 至少一个学生学习任务可由多个 agent 串联完成
- 上一个 agent 的结果能成为下一个 agent 的输入
- 最终结果能反馈回学习路径或教师端

#### A3. 必须围绕真实任务工作

通过标准：

- AI 输出要绑定教材、任务、错题、评测等真实学习对象
- 不是脱离上下文的泛化闲聊
- 至少有一条“诊断 -> 执行 -> 反馈”的可复现场景

#### A4. 必须具备可解释性

通过标准：

- 推荐结果可以说明依据
- 评测结果可以说明原因
- 教师建议可以说明触发条件

### 6.2 B 类标准：建议项

#### B1. 应具备服务端学习记忆

通过标准：

- 学习状态不只保存在本地浏览器
- agent 可以跨设备延续学习档案
- 教师可看到统一学习视图

#### B2. 应具备动态任务调整能力

通过标准：

- 学生完成一次评测后，下一步推荐会改变
- 学生出现连续错误时，会自动提高复习优先级
- 学生表现进步时，会自动进入更高阶训练

#### B3. 应具备教师侧 agent 输出

通过标准：

- 教师端不只是看原始数据
- 系统可以自动输出班级洞察或干预建议
- 可以标记高风险学生或重点关注对象

### 6.3 C 类标准：加分项

#### C1. 多 agent 决策链可追踪

通过标准：

- 每次推荐、评测和建议都有日志
- 可以追溯由谁触发、依据什么状态、产出什么结果

#### C2. 支持多层级策略

通过标准：

- 年级不同，策略不同
- 能力水平不同，训练路径不同
- 教师和学生看到的结果层次不同

#### C3. 可量化业务价值

通过标准：

- 能统计任务完成率、错题回练率、发音分提升
- 能衡量教师节省时间或干预效率

## 7. 多 Agent 验收场景脚本

以下脚本用于现场演示或评审复现。

### 场景一：单元学习闭环

目标：

验证多个 agent 能否围绕一个教材单元形成闭环。

步骤：

1. 学生进入某教材单元
2. 学习编排 Agent 推荐先做词汇预习或重点句训练
3. 学生进入口语或听写练习
4. 口语陪练 Agent 或评测反馈 Agent 给出结果
5. 学习编排 Agent 根据结果更新下一步建议

通过判定：

- 不同 agent 的输出能接续
- 推荐内容发生变化且有依据
- 学生看到的不是孤立功能点，而是一条学习链路

### 场景二：口语练习与发音评测联动

目标：

验证口语陪练 agent 与评测反馈 agent 是否协同。

步骤：

1. 学生发起口语练习
2. 口语陪练 Agent 给出对练内容
3. 学生完成朗读或表达
4. 评测反馈 Agent 输出发音分、薄弱词和建议
5. 学习编排 Agent 生成下一步巩固任务

通过判定：

- 口语训练与评测结果共享上下文
- 评测结果影响后续任务安排

### 场景三：教师干预闭环

目标：

验证学生侧 agent 结果是否能被教师侧 agent 消费。

步骤：

1. 学生连续出现错题或发音问题
2. 评测反馈 Agent 标记问题类型
3. 教师洞察 Agent 汇总风险信号
4. 教师端展示风险学生与建议动作

通过判定：

- 教师端不是只显示原始数据
- 存在可解释的风险识别与干预建议

## 8. 当前版本建议重点补强项

从 AI agent 落地验收角度，建议优先增强以下能力。

### 8.1 必须优先补强

1. 明确 agent 分层与命名
2. 增加统一的学习编排 agent
3. 建立服务端学习记忆
4. 让推荐、评测、教师建议形成可解释决策链
5. 让评测结果真正影响下一步学习路径

### 8.2 第二优先级补强

1. 口语陪练从自由聊天升级为目标导向任务对练
2. 错题与发音问题形成统一的复盘包
3. 教师端自动生成班级周报与风险建议
4. 增加 agent 日志与运行记录

### 8.3 第三优先级补强

1. 引入分层教学策略
2. 引入效果指标体系
3. 引入更多教材内容与内容运营工作流

## 9. 建议验收结论模板

评审结束后，可以使用以下模板输出结论。

### 9.1 通过

> 该应用已经具备 AI agent 落地应用的基本特征。系统中存在明确的多 agent 分工，并已在教材学习、口语训练、评测反馈和教师洞察之间形成协同闭环。建议作为垂直教育场景的 AI agent 实践案例继续推进。

### 9.2 有条件通过

> 该应用已经具备多 agent 协同雏形，能够展示 AI 在英语学习场景中的任务化落地能力。但服务端记忆、任务编排和教师洞察链路仍需补强。建议作为 MVP 进入下一阶段优化，而非直接按成熟产品对外承诺。

### 9.3 不通过

> 该应用当前仍以单点 AI 能力调用为主，尚未形成清晰的多 agent 分工与协同闭环。建议先补齐学习编排、结果回流和可解释性，再重新发起 AI agent 应用验收。
