# English Assist 应用评审与增强清单

## 结论

这个应用已经具备拿来讲解 AI agent 落地实践的基础。

它不是单纯聊天 demo，而是把教材、学习路径、语音评测、听写训练、教师管理和云端部署串成了一个完整场景，适合对外展示“AI 如何进入垂直教育流程”。

但如果要把它作为面向他人的推荐案例，当前更适合定位成：

- 可演示的垂直教育 AI 原型
- 可以验证价值的教学场景 MVP

还不适合直接包装成“可规模推广的成熟产品”，主要原因是安全、数据持久化、评测可信度和运营可控性还不够。

## 适合推荐的亮点

- 教材同步学习路径清晰：把预习、重点句、语法、听写、口语评测串成单元闭环。
- AI 能力不是悬浮功能：AI 对话、TTS、Azure Speech 发音评测都嵌入了学习任务。
- 教学产品感较完整：学生端、教师管理端、教材导入、统计面板、Vercel + Turso 部署链路都已具备。
- 技术栈轻量：React + Express + libSQL/Turso，便于复刻和二次开发。
- 适合做“场景化 AI 应用”示例：比通用聊天机器人更能说明业务落地。

## 当前不够好的地方

### P0：安全与上线风险

1. 默认管理员账号密码写死在代码里，且部署文档鼓励上线后直接使用。
2. AI API Key 存在浏览器本地，并由前端直接请求模型服务，无法做统一鉴权、审计、限流和成本控制。
3. 缺少登录风控、限流、操作审计和敏感行为保护，不适合直接面向真实学校或机构用户开放。

### P1：产品可信度与数据闭环

1. 学习记录保存在浏览器 localStorage，不是服务端真实学习档案。
2. 教师管理页读取的是本地快照，不是后端采集到的统一班级数据。
3. 换设备、清缓存、跨浏览器后学习进度可能丢失，教师视角与学生真实行为也可能不一致。

### P1：AI 评测与结果说服力不足

1. 单元闯关当前按“是否填写内容”计分，不校验答案质量。
2. AI 对话虽可用，但没有课程目标、回合策略、难度分层和纠错策略闭环。
3. 缺少可解释的学习建议生成逻辑，当前推荐更像规则驱动而不是真正的数据驱动 agent。

### P2：工程化与运营能力不足

1. 缺少 migration、seed 版本管理、监控告警、错误追踪和行为埋点。
2. 缺少测试覆盖，难以证明回归稳定性。
3. 教材导入仍偏手工样板，不是完整 CMS 或内容工作流。

## 增强清单

### 第一阶段：把它从“可演示原型”升到“可推荐方案”

1. 移除默认管理员口令，首启时强制初始化管理员密码。
2. 把 AI 调用改成服务端代理，前端不再直接持有供应商 API Key。
3. 把学习进度、错题、发音评测、闯关记录迁移到数据库。
4. 教师管理页改为读取后端真实班级数据，而不是本地浏览器快照。
5. 给登录、教材导入、管理接口增加限流和操作审计。
6. 增加基础错误监控和接口日志，至少覆盖登录、教材读取、AI 调用、Speech token 和导入流程。

### 第二阶段：把它从“推荐方案”升到“AI 教育产品”

1. 重做单元闯关评分逻辑，加入标准答案比对、关键词校验和 AI 评分 rubric。
2. 给 AI 对话增加任务状态机，支持目标驱动、分层提示、追问和复盘。
3. 生成个性化学习计划，把错题、发音薄弱词、学习频次和教材进度统一进推荐逻辑。
4. 为教师提供班级画像、风险学生识别、阶段报告和干预建议。
5. 建立教材内容生产后台，支持样板导入、审核、版本化和发布。

### 第三阶段：把它包装成“AI agent 落地实践案例”

1. 明确 agent 定位：不是聊天助手，而是“学习编排 agent + 语音评测 agent + 教师洞察 agent”。
2. 用真实指标讲故事：完课率、连续学习天数、错题回练率、发音分提升、教师节省的批改时间。
3. 增加观察面板：agent 决策日志、推荐依据、模型调用量、失败率、平均响应时长。
4. 补一套案例素材：典型学生路径、教师使用流程、部署架构图、ROI 说明。

## 对外推荐时的建议话术

建议你对外把它描述为：

“一个面向初高中英语场景的 AI 学习助教原型，已经把教材同步学习、语音评测、AI 对话练习和教师管理串起来，适合作为垂直教育 AI agent 落地实践案例。当前已具备完整演示闭环，下一步重点是把安全、数据持久化和评测可信度补齐，升级为可规模化交付的产品方案。”

这个表述既真实，也有说服力。
